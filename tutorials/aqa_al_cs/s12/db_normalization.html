4.10.3 Database Design and Normalisation Techniques

An interactive guide to understanding database normalisation.

Why Normalise Databases?

Database normalisation is the process of structuring a relational database to reduce data redundancy and improve data integrity. It was first proposed by Edgar F. Codd in the 1970s to prevent undesirable dependencies between data. The main goals are to eliminate redundant data, minimize errors when modifying data, and simplify the process of querying the database.

Key Objectives of Normalisation:

  • Reduce Data Redundancy: Avoids storing the same information in multiple places, which saves storage space and prevents inconsistencies.
  • Improve Data Integrity: Ensures data is reliable and accurate. When data is only stored once, updates are simpler and less prone to errors.
  • Prevent Data Anomalies: Helps to avoid issues with inserting, updating, and deleting data. This includes insertion, deletion, and update anomalies.
  • Simplify Database Maintenance: A well-normalised database is easier to manage and maintain.

Normalising Relations to Third Normal Form (3NF)

To reach Third Normal Form (3NF), a relation must first be in First Normal Form (1NF) and Second Normal Form (2NF).

First Normal Form (1NF)

A relation is in 1NF if all its attributes contain only atomic (indivisible) values, and each record is unique.

Un-normalised Table:

StudentID Name Courses
1 Alice History, Maths
2 Bob Physics

Table in 1NF:

StudentID Name Course
1 Alice History
1 Alice Maths
2 Bob Physics

Second Normal Form (2NF)

A relation is in 2NF if it is in 1NF and all non-key attributes are fully functionally dependent on the entire primary key. This is primarily relevant for tables with composite primary keys.

Table not in 2NF (Composite Primary Key: StudentID, Course):

StudentID Course Instructor
1 History Prof. Jones
1 Maths Prof. Davis
2 History Prof. Jones

Here, `Instructor` only depends on `Course`, which is only part of the primary key. This is a partial dependency.

Tables in 2NF:

Enrollment Table

StudentID Course
1 History
1 Maths
2 History

Courses Table

Course Instructor
History Prof. Jones
Maths Prof. Davis

Third Normal Form (3NF)

A relation is in 3NF if it is in 2NF and has no transitive dependencies. This means that no non-key attribute is dependent on another non-key attribute.

Table not in 3NF (from our 2NF example):

Course Instructor Instructor_Department
History Prof. Jones Humanities
Maths Prof. Davis Science
Physics Prof. Maxwell Science

Here, `Instructor_Department` depends on `Instructor`, which is a non-key attribute. This is a transitive dependency.

Tables in 3NF:

Courses Table

Course Instructor_ID
History 101
Maths 102
Physics 103

Instructors Table

Instructor_ID Instructor_Name Department
101 Prof. Jones Humanities
102 Prof. Davis Science
103 Prof. Maxwell Science

Properties of a Relation in Third Normal Form

A relation in 3NF has specific characteristics that contribute to a well-structured database.

Key Properties:

  • It is in Second Normal Form (2NF).
  • It has no transitive dependencies. This means that every non-prime attribute is non-transitively dependent on every candidate key.
  • For any functional dependency X → Y, either X is a superkey, or Y is a prime attribute (part of a candidate key).
  • Most tables in 3NF are free of update, insertion, and deletion anomalies.
