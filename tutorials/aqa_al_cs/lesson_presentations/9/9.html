Why is the Object-Oriented Paradigm Used?

Better Real-World Modelling: Objects can naturally represent real-world entities (e.g., a Student object, a Course object).
Improved Modularity & Maintainability: Encapsulation means changes within a class are less likely to break other parts of the system.
Enhanced Reusability: Inheritance and composition allow classes and objects to be reused and extended in different projects.
Increased Extensibility & Scalability: New features can be added by creating new classes without altering existing, working code.
Data Hiding & Security: Encapsulation protects an object's internal state from unintended external modification.
Collaboration: Clear class interfaces facilitate teamwork, as developers can work on different classes independently.
Polymorphism for Flexibility: Allows for treating different objects uniformly, leading to more flexible and adaptable code.

Principle 1: Encapsulate What Varies

Identify the parts of your application that are most likely to change and separate them from the parts that stay the same.

Example: Regional Tax Calculation

Instead of scattering IF-ELSE statements for different regions throughout your code, encapsulate the tax calculation logic.

  • Create a `TaxCalculator` interface.
  • Implement `UKTaxCalculator`, `EUTaxCalculator`, etc.
  • The rest of the system uses the `TaxCalculator` interface, unaware of the specific regional implementation.

Supported by: Encapsulation, Interfaces, Abstract Classes.

Python Example: A Payment Processing System

Imagine you need to process payments, but might add new payment methods later (e.g., Bank Transfer, Bitcoin). The payment method is what varies.

Before 


class Order:
    def __init__(self, amount, payment_method):
        self.amount = amount
        self.payment_method = payment_method

    def process_payment(self):
        if self.payment_method == "credit_card":
            print(f"Paying £{self.amount} using Credit Card.")
        elif self.payment_method == "paypal":
            print(f"Paying £{self.amount} via PayPal.")
        # To add new methods, you need to keep modifying this method:
        # elif self.payment_method == "bank_transfer":
        #     print(f"Paying £{self.amount} via Bank Transfer.")
        else:
            print("Unknown payment method.")

# Usage
order1 = Order(100, "credit_card")
order1.process_payment()

order2 = Order(50, "paypal")
order2.process_payment()

After 


from abc import ABC, abstractmethod

# 1. Define a stable interface (the part that doesn't change)
class PaymentProcessor(ABC):
    @abstractmethod
    def pay(self, amount):
        pass

# 2. Encapsulate the varying parts (the implementations)
class CreditCardProcessor(PaymentProcessor):
    def pay(self, amount):
        print(f"Paying £{amount} using Credit Card.")

class PayPalProcessor(PaymentProcessor):
    def pay(self, amount):
        print(f"Paying £{amount} via PayPal.")

# 3. The client code depends only on the stable interface
class Order:
    def __init__(self, amount, processor: PaymentProcessor):
        self.amount = amount
        self.processor = processor

    def process_payment(self):
        self.processor.pay(self.amount)

# You can easily switch the implementation without changing the Order class
cc_processor = CreditCardProcessor()
order1 = Order(100, cc_processor)
order1.process_payment()

paypal_processor = PayPalProcessor()
order2 = Order(50, paypal_processor)
order2.process_payment()

Principle 2: Favour Composition Over Inheritance

Build complex objects by assembling other objects, rather than inheriting from a complex base class.

Inheritance ("is-a")

Creates strong coupling. Changes to the superclass can break subclasses. Can lead to rigid hierarchies.

Composition ("has-a")

More flexible. Objects can be combined at runtime. Avoids the "Fragile Base Class" problem.

When to use inheritance: When there's a clear "is-a" relationship and you want to leverage polymorphism with a common interface defined by the superclass. It should model a genuine specialization.

Python Example: A Car with an Engine

A Car has an Engine (composition) is more flexible than Car is an EnginePoweredVehicle (inheritance), especially if you later want an ElectricCar which has a Motor.

Before 


# The base class for all engine-powered vehicles
class EnginePoweredVehicle:
    def start(self):
        return "Generic engine sound"

# A Car is an EnginePoweredVehicle with a petrol engine
class Car(EnginePoweredVehicle):
    def start(self):
        return "Vroom!"

# An ElectricCar is an EnginePoweredVehicle with an electric motor
class ElectricCar(EnginePoweredVehicle):
    def start(self):
        return "Whirrr..."

# Create instances and start them
petrol_car = Car()
print(f"Starting the petrol car... {petrol_car.start()}")

electric_car = ElectricCar()
print(f"Starting the electric car... {electric_car.start()}")

After


# The "part" objects
class Engine:
    def start(self):
        return "Vroom!"

class ElectricMotor:
    def start(self):
        return "Whirrr..."

# The "whole" object, built by composition
class Car:
    def __init__(self, engine_type):
        # The Car "has-a" engine, which is another object
        self.engine = engine_type

    def start_car(self):
        print(f"Starting the car... {self.engine.start()}")

# Create cars with different "parts" at runtime
petrol_engine = Engine()
petrol_car = Car(petrol_engine)
petrol_car.start_car()

electric_motor = ElectricMotor()
electric_car = Car(electric_motor)
electric_car.start_car()

Principle 3: Program to Interfaces, Not Implementation

Depend on abstractions (interfaces or abstract classes) rather than concrete classes.

  • Interface: Defines a contract (a set of methods a class must implement) without providing the implementation.
  • Implementation: The concrete class that provides the actual code for the methods defined in an interface.

Purpose: Promotes loose coupling. Your code interacts with an abstract contract, not a specific type. This makes it easy to swap out different implementations without changing the client code.

Example:

INTERFACE SwitchableDevice:
    METHOD turnOn()
    METHOD turnOff()

CLASS LightBulb IMPLEMENTS SwitchableDevice: ...
CLASS Fan IMPLEMENTS SwitchableDevice: ...

// Client code:
device AS SwitchableDevice // Program to the interface
device = new LightBulb()
device.turnOn()

The client code doesn't care if it's a LightBulb or a Fan, only that it's a SwitchableDevice and can be turned on.

Python Code:


from abc import ABC, abstractmethod

# Define the interface
class SwitchableDevice(ABC):
    @abstractmethod
    def turnOn(self):
        pass

    @abstractmethod
    def turnOff(self):
        pass

# Implement the LightBulb class
class LightBulb(SwitchableDevice):
    def turnOn(self):
        print("LightBulb is now ON.")

    def turnOff(self):
        print("LightBulb is now OFF.")

# Implement the Fan class
class Fan(SwitchableDevice):
    def turnOn(self):
        print("Fan is now ON.")

    def turnOff(self):
        print("Fan is now OFF.")

# Client code: program to the interface
device: SwitchableDevice

device = LightBulb()
device.turnOn()  # Output: LightBulb is now ON.

Coding: Access Specifiers

  • Public (+): Accessible from anywhere.
  • Private (-): Accessible only from within the class itself. Key for encapsulation.
  • Protected (#): Accessible within the class and its subclasses.

Example (Python convention):

class BankAccount:
    def __init__(self, initial_balance):
        self.__balance = initial_balance  # Private attribute
        self.account_holder = "Default" # Public attribute

    def deposit(self, amount): # Public method
        if amount > 0:
            self.__balance += amount

    def get_balance(self): # Public "getter" method
        return self.__balance

Conceptual: Advanced Method Types

Abstract Method

A method declared in an abstract class with no implementation. Subclasses are forced to provide an implementation. Defines a contract (e.g., `calculateArea()` in a `Shape` class).

Virtual Method

A method in a base class that can be overridden by a derived class. Enables polymorphic behavior. (In Python, all methods are effectively virtual).

Static Method

Belongs to the class, not an instance. Called using the class name (e.g., `MathUtils.add(5, 3)`). Used for utility functions.

Class Diagrams: Relationships

Inheritance ("is-a")

Arrow with an open arrowhead from subclass to superclass. subclass ───▷ superclass

Aggregation ("has-a", shared)

Line with an open diamond on the "whole" class side. whole ◇───── part

Composition ("has-a", owned)

Line with a filled diamond on the "whole" class side. whole ◆───── part

Summary & Key Takeaways

  • OOP is used for better modelling, maintainability, and reusability.
  • Key Principles: Encapsulate what varies, favour composition, program to interfaces.
  • Practical OOP involves classes, attributes, methods, and access specifiers.
  • Class diagrams are vital for designing and visualizing system structure and relationships.
